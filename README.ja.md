# batch-scheduler

[English](README.md) | **日本語**

小さく堅牢な MVP のバッチ実行マネージャーです。YAML ファイルにジョブの集合を
定義すると、依存関係の DAG（有向非巡回グラフ）を尊重してジョブを実行します。
各ジョブは設定に応じてリトライやタイムアウトが行われ、失敗したジョブに依存する
ものはスキップされ、すべての実行は JSON レコードとして永続化されるため、後から
履歴と結果を確認できます。

## デモ

代表フロー `examples/etl.yaml`（extract → transform → load → notify）の端末録画です。
バッチ定義の読み込み → 依存グラフ（DAG）の検証 → トポロジカル順での実行 →
実行履歴の一覧表示までを示します。

![examples/etl.yaml のバッチ定義を表示し、DAG を検証してからトポロジカル順に実行し、実行履歴を一覧表示する端末操作のデモ](docs/demo.gif)

## 特徴

- **DAG 依存関係** — ジョブは `dependsOn` を宣言し、検証済みのトポロジカル順序で
  実行されます。
- **リトライ** — ジョブは最初の失敗後に、設定した回数だけ再試行できます。
- **ジョブごとのタイムアウト** — 各試行は秒単位のタイムアウトで制限できます。
- **失敗時の依存ジョブのスキップ** — ジョブが `FAILED`（または自身がスキップ）に
  なると、それに推移的に依存するすべてのジョブが `SKIPPED` とマークされ、実行全体は
  `FAILED` として報告されます。
- **JSON での実行履歴の永続化** — 各実行は JSON ドキュメントとして保存されるため、
  過去の実行とその結果を一覧できます。

## 必要要件

- Java 21
- Maven 3.9+

## ビルド

```sh
mvn package
```

これにより、実行可能な shaded jar が `target/batch-scheduler.jar` に生成されます。

## 使い方

```sh
# バッチを実行する（ジョブを実行して実行結果を記録する）
java -jar target/batch-scheduler.jar run examples/etl.yaml

# 実行せずにバッチを検証する（DAG をチェックし、実行順序を表示する）
java -jar target/batch-scheduler.jar validate examples/etl.yaml

# 過去に記録された実行を一覧する（新しい順）
java -jar target/batch-scheduler.jar list
```

便利なオプション:

- `run --state-dir <dir>` / `list --state-dir <dir>` — 実行状態を保存する
  ディレクトリ（デフォルト: `.batch-state`）。
- `run -q` / `run --quiet` — ジョブごとのサマリーテーブルの表示を抑制する。
- `run --rerun-failed <runId>` — 前回実行（`--state-dir` 配下から runId で検索）で
  `SUCCEEDED` だったジョブの結果を流用し、`FAILED`/`SKIPPED` だったジョブと前回に
  存在しなかった新規ジョブだけを（再）実行する。詳細は
  [失敗したジョブだけを再実行する](#失敗したジョブだけを再実行する) を参照。
- `list -n <n>` / `list --limit <n>` — 最新の実行を最大 `n` 件だけ表示する
  （デフォルト: `20`）。`0` または負の値を渡すと全件を表示する。
- `--help`、`--version` — 標準のヘルプとバージョン出力（各サブコマンドでも利用可能）。

### 失敗したジョブだけを再実行する

`run` の結果、一部のジョブが `FAILED` や `SKIPPED` で終わった場合、原因を修正した
うえで、既に成功したジョブをやり直すことなく、それらのジョブ（および前回以降に
バッチ定義へ新規追加されたジョブ）だけを再実行できます:

```sh
# 失敗した実行が表示した「Run ID」を控えておく（または `list` で確認する）。
java -jar target/batch-scheduler.jar run examples/etl.yaml --rerun-failed 20260101-120000-abc123def456
```

前回 `SUCCEEDED` だったジョブはそのまま流用されます（前回の結果が新しい実行結果へ
コピーされ、再実行はされません）。前回 `FAILED`/`SKIPPED` だったジョブ、および
バッチ定義には存在するが前回結果には無かったジョブは通常どおり実行されます。
指定する runId は同じ `--state-dir` 配下に存在している必要があり、かつ同じ `name`
を持つバッチの実行記録でなければなりません（別バッチの runId を指定した場合は
無関係なジョブの結果を紛れ込ませず、設定・IO エラー（終了コード `3`）として拒否
されます）。これは「再開（resume）」ではなく「再実行（rerun）」であり、常に
新しい runId が発行され、バッチ内の全ジョブを含む新しい実行記録が保存されます。

`--rerun-failed` は、`SUCCEEDED` だったジョブ自体の定義（`command` や `dependsOn`
等）が前回実行時から変更されていないかまでは検証しません。バッチ定義の他の部分と
同様、どのジョブを再実行せず流用してよいかは実行者の判断に委ねられます。

## YAML スキーマ

バッチファイルには `name` と `jobs` のリストがあります。各ジョブは次の
フィールドで定義されます:

| フィールド        | 型                | 必須 | デフォルト       | 説明 |
|------------------|-------------------|------|----------------|------|
| `id`             | 文字列            | はい | —              | ジョブの一意な識別子。 |
| `name`           | 文字列            | いいえ | `id` の値      | 人間が読みやすいラベル。 |
| `command`        | 文字列の配列      | はい | —              | 実行するコマンド。**argv 配列**（プログラムとその引数）として指定。 |
| `dependsOn`      | 文字列の配列      | いいえ | `[]`           | このジョブを実行する前に成功している必要があるジョブの id。 |
| `retries`        | 整数 (>= 0)       | いいえ | `0`            | 最初の失敗後に行う*追加の*試行回数。 |
| `timeoutSeconds` | 整数 (>= 0)       | いいえ | `0`（無制限）  | 試行ごとのタイムアウト（秒）。 |
| `env`            | 文字列→文字列のマップ | いいえ | `{}`         | 生成されるプロセスに追加する環境変数。 |
| `workingDir`     | 文字列            | いいえ | 起動元のディレクトリ | プロセスの作業ディレクトリ。 |

> 注意: `command` は**argv 配列**であり、シェル文字列ではありません。シェルの機能
> （パイプ、グロブ、`&&` など）を使うには、明示的にシェルを呼び出してください。
> 例: `["sh", "-c", "echo hi && ls"]`。

### サンプル

```yaml
name: etl

jobs:
  - id: extract
    name: Extract raw data
    command: ["sh", "-c", "echo extract"]

  - id: transform
    name: Transform extracted data
    command: ["sh", "-c", "echo transform"]
    dependsOn: [extract]
    retries: 2
    timeoutSeconds: 30

  - id: load
    command: ["sh", "-c", "echo load"]
    dependsOn: [transform]
    workingDir: /tmp
    env:
      STAGE: load
```

## 終了コード

| コード | 意味 |
|------|------|
| `0`  | 成功 — 操作が完了した（`run` の場合はバッチが `SUCCEEDED`）。 |
| `1`  | バッチは最後まで実行されたが、`FAILED` 状態で終了した。 |
| `2`  | 検証エラー — バッチの構造が不正（不正な DAG、依存先の欠落、id の重複、空または空白のみのコマンドトークン、循環）。 |
| `3`  | 設定 / IO エラー（ファイルが存在しない、解析できない、または実行前に `--state-dir` を準備できない）、または使用方法のエラー。 |

補足:

- `run` は状態ディレクトリの準備（終了コード `3`）より先にバッチを検証（終了
  コード `2`）するため、無効なバッチが `--state-dir` のディレクトリツリーを
  副作用として作ることはありません。
- 実行**後**の実行記録の保存に失敗した場合（例: 実行中に状態ディレクトリが
  書き込めなくなった）は、バッチの結果が優先されます。失敗したバッチの後は
  終了コード `1` のままで、成功したバッチの後にのみ保存失敗が終了コード `3`
  として報告されます。
- `timeoutSeconds` と `retries` は整数のみを受け付けます。`30.0` のような
  整数値の小数表記を含め、すべての小数リテラルは切り捨てられずに設定エラー
  （終了コード `3`）として拒否されます。

## セキュリティと信頼モデル

バッチファイルは**信頼された入力**です。`Makefile` や CI パイプラインと同様に、
設計上、任意のコマンドを実行できるため、信頼できないソースのバッチファイルは
実行しないでください。そのモデルの範囲内でも、本ツールはリソースの枯渇（境界の
ある YAML 解析、境界のあるジョブ出力のキャプチャ、反復的なグラフアルゴリズム）や、
状態ディレクトリからの脱出（run-id の検証、シンボリックリンクを辿らない）に対する
防御を行います。詳細は [docs/DESIGN.md](docs/DESIGN.md#security--trust-model) を
参照してください。
