# batch-scheduler

[English](README.md) | **日本語**

小さく堅牢な MVP のバッチ実行マネージャーです。YAML ファイルにジョブの集合を
定義すると、依存関係の DAG（有向非巡回グラフ）を尊重してジョブを実行します。
各ジョブは設定に応じてリトライやタイムアウトが行われ、失敗したジョブに依存する
ものはスキップされ、すべての実行は JSON レコードとして永続化されるため、後から
履歴と結果を確認できます。

## 特徴

- **DAG 依存関係** — ジョブは `dependsOn` を宣言し、検証済みのトポロジカル順序で
  実行されます。
- **リトライ** — ジョブは最初の失敗後に、設定した回数だけ再試行できます。
- **ジョブごとのタイムアウト** — 各試行は秒単位のタイムアウトで制限できます。
- **失敗時の依存ジョブのスキップ** — ジョブが `FAILED`（または自身がスキップ）に
  なると、それに推移的に依存するすべてのジョブが `SKIPPED` とマークされ、実行全体は
  `FAILED` として報告されます。
- **JSON での実行履歴の永続化** — 各実行は JSON ドキュメントとして保存されるため、
  過去の実行とその結果を一覧できます。

## 必要要件

- Java 21
- Maven 3.9+

## ビルド

```sh
mvn package
```

これにより、実行可能な shaded jar が `target/batch-scheduler.jar` に生成されます。

## 使い方

```sh
# バッチを実行する（ジョブを実行して実行結果を記録する）
java -jar target/batch-scheduler.jar run examples/etl.yaml

# 実行せずにバッチを検証する（DAG をチェックし、実行順序を表示する）
java -jar target/batch-scheduler.jar validate examples/etl.yaml

# 過去に記録された実行を一覧する（新しい順）
java -jar target/batch-scheduler.jar list
```

便利なオプション:

- `run --state-dir <dir>` / `list --state-dir <dir>` — 実行状態を保存する
  ディレクトリ（デフォルト: `.batch-state`）。
- `run -q` / `run --quiet` — ジョブごとのサマリーテーブルの表示を抑制する。
- `--help`、`--version` — 標準のヘルプとバージョン出力（各サブコマンドでも利用可能）。

## YAML スキーマ

バッチファイルには `name` と `jobs` のリストがあります。各ジョブは次の
フィールドで定義されます:

| フィールド        | 型                | 必須 | デフォルト       | 説明 |
|------------------|-------------------|------|----------------|------|
| `id`             | 文字列            | はい | —              | ジョブの一意な識別子。 |
| `name`           | 文字列            | いいえ | `id` の値      | 人間が読みやすいラベル。 |
| `command`        | 文字列の配列      | はい | —              | 実行するコマンド。**argv 配列**（プログラムとその引数）として指定。 |
| `dependsOn`      | 文字列の配列      | いいえ | `[]`           | このジョブを実行する前に成功している必要があるジョブの id。 |
| `retries`        | 整数 (>= 0)       | いいえ | `0`            | 最初の失敗後に行う*追加の*試行回数。 |
| `timeoutSeconds` | 整数 (>= 0)       | いいえ | `0`（無制限）  | 試行ごとのタイムアウト（秒）。 |
| `env`            | 文字列→文字列のマップ | いいえ | `{}`         | 生成されるプロセスに追加する環境変数。 |
| `workingDir`     | 文字列            | いいえ | 起動元のディレクトリ | プロセスの作業ディレクトリ。 |

> 注意: `command` は**argv 配列**であり、シェル文字列ではありません。シェルの機能
> （パイプ、グロブ、`&&` など）を使うには、明示的にシェルを呼び出してください。
> 例: `["sh", "-c", "echo hi && ls"]`。

### サンプル

```yaml
name: etl

jobs:
  - id: extract
    name: Extract raw data
    command: ["sh", "-c", "echo extract"]

  - id: transform
    name: Transform extracted data
    command: ["sh", "-c", "echo transform"]
    dependsOn: [extract]
    retries: 2
    timeoutSeconds: 30

  - id: load
    command: ["sh", "-c", "echo load"]
    dependsOn: [transform]
    workingDir: /tmp
    env:
      STAGE: load
```

## 終了コード

| コード | 意味 |
|------|------|
| `0`  | 成功 — 操作が完了した（`run` の場合はバッチが `SUCCEEDED`）。 |
| `1`  | バッチは最後まで実行されたが、`FAILED` 状態で終了した。 |
| `2`  | 検証エラー — バッチの構造が不正（不正な DAG、依存先の欠落、id の重複、空のコマンド、循環）。 |
| `3`  | 設定 / IO エラー（ファイルが存在しない、または解析できない）、または使用方法のエラー。 |

## セキュリティと信頼モデル

バッチファイルは**信頼された入力**です。`Makefile` や CI パイプラインと同様に、
設計上、任意のコマンドを実行できるため、信頼できないソースのバッチファイルは
実行しないでください。そのモデルの範囲内でも、本ツールはリソースの枯渇（境界の
ある YAML 解析、境界のあるジョブ出力のキャプチャ、反復的なグラフアルゴリズム）や、
状態ディレクトリからの脱出（run-id の検証、シンボリックリンクを辿らない）に対する
防御を行います。詳細は [docs/DESIGN.md](docs/DESIGN.md#security--trust-model) を
参照してください。
